# Decision 0001: 代码仓库托管策略(GitHub / Codeup)

**状态**: 待决策(open)

**最后更新**: 2026-04-22

## 背景

项目 `wirachat-h5-starter` 目前托管在 GitHub(`git@github.com:YoungBlackBlack/wirachat-h5-starter.git`),分支 `main`。

即将开始多人协作,需要决策代码仓库托管策略。候选平台:

- **GitHub** — 当前托管方,国际生态成熟,国内访问偶发慢
- **阿里云 Codeup** — 国内速度好,与阿里云 DevOps(云效)/ECS/函数计算深度集成

- [ ] 是否需要 **PR review** 流程?(两平台都支持,GitHub UI 更成熟)

## 候选方案对比

### 方案 1: Codeup 主,GitHub 不用

- 所有人只在 Codeup 协作,GitHub 仓库归档或删除
- **优点**: 零同步成本,国内体验最好
- **缺点**: 失去 GitHub 生态(第三方 CI、社区能见度)
- **适用**: 国内团队 + 不开源 + 用阿里云

### 方案 2: GitHub 主,Codeup 只读镜像(单向同步)

- GitHub 做协作主仓库(PR / review / merge)
- Codeup 做镜像,仅供部署机拉代码
- 同步方向: **GitHub → Codeup**(Codeup 原生支持定时拉取,或用 GitHub Actions 推)
- **优点**: 保留 GitHub 生态 + 阿里云部署友好
- **缺点**: 有延迟(原生镜像模式);需维护同步配置
- **适用**: 有海外成员,或想保留 GitHub 工作流,但部署在阿里云

### 方案 3: Codeup 主,GitHub 展示镜像(反向单向同步)

- 内部协作全在 Codeup
- GitHub 仅做对外展示 / 开源版本
- 同步方向: **Codeup → GitHub**
- **优点**: 公司代码在 Codeup(合规) + 维护开源形象
- **缺点**: Codeup → GitHub 不原生,需要 CI 脚本维护
- **适用**: 公司强制 Codeup 但想做开源展示

### 方案 4: 双主(两边都能 push)

- **强烈不推荐**
- 冲突是必然不是偶然
- 除非有专职 release 工程师管理仓库

## 暂定建议(待问题回答后确认)

**如果** 5 个问题的答案倾向于:
- 国内团队 + 用阿里云部署 + 不开源 → **方案 1**(Codeup 单主)
- 有海外成员或想保留 GitHub 工作流 + 部署在阿里云 → **方案 2**(GitHub 主 + Codeup 镜像)
- 其他组合 → 讨论后定

**通用建议(不论选哪个方案)**:

1. **先定主仓库**,不要两边同时写
2. **分支保护**必须开:`main` 禁止直接 push,只能 PR 合入
3. **秘钥别进仓库** — `.env` / `*.secret` 必须 gitignore
4. **CHANGELOG 风格统一** — 继续 Keep a Changelog,每次会话一节
5. **部署机用 deploy key** — 只读 SSH key,不要用个人账号凭证
